# Architecture

## 系统架构 / System Architecture

本项目采用分层架构设计，遵循Clean Architecture原则。

### 架构图

```
┌─────────────────────────────────────────────────┐
│                   Presentation Layer            │
│              (cmd/api, internal/api)           │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│                   Business Logic Layer          │
│               (internal/service)                │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│                   Data Access Layer             │
│              (internal/repository)              │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│                   Database                      │
└─────────────────────────────────────────────────┘
```

## 层次说明

### Presentation Layer (表现层)
- 处理HTTP请求和响应
- 参数验证
- 认证和授权
- 中间件处理

### Business Logic Layer (业务逻辑层)
- 核心业务规则
- 业务流程编排
- 领域模型操作

### Data Access Layer (数据访问层)
- 数据库操作
- 数据持久化
- 查询优化

## 设计原则

1. **依赖倒置**: 高层模块不依赖低层模块，都依赖抽象
2. **单一职责**: 每个模块只负责一个功能领域
3. **开闭原则**: 对扩展开放，对修改关闭
4. **接口隔离**: 使用小而专的接口

## 技术决策

### 为什么选择这种结构？

- **可维护性**: 清晰的层次划分使代码易于维护
- **可测试性**: 各层独立，便于单元测试
- **可扩展性**: 模块化设计支持功能扩展
- **团队协作**: 结构清晰，便于团队分工
